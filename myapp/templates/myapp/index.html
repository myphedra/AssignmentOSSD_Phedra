Final Assignment Documentation

This page presents the theoretical foundations and system design of the final assignment for the Open Source Software Development course at Saigon University. The project implements a Retrieval-Augmented Generation (RAG) system for intelligent PDF document question answering, supporting Vietnamese and English with fully local, privacy-preserving inference powered entirely by open-source tools.

01

Introduction

1.1 Background and Motivation

In the current digital era, the volume of information stored inside electronic documents is growing at an unprecedented rate. Efficiently searching for and extracting relevant knowledge from such documents has become one of the most pressing challenges in applied Natural Language Processing (NLP). Traditional retrieval methods — such as keyword-based search — are fundamentally limited: they match surface-level terms rather than understanding the true semantic intent behind a user's query, frequently returning irrelevant results or missing critical information expressed in different words.

The RAG (Retrieval-Augmented Generation) paradigm was developed to address exactly this gap. By combining two complementary capabilities, RAG systems can understand nuanced questions and return contextually grounded, human-readable answers:

🔍
Information Retrieval

Semantically locate and rank the most relevant passages from a structured knowledge base using dense vector similarity search, going far beyond keyword overlap.

✍️
Natural Language Generation

Produce fluent, factually grounded, and context-aware answers by feeding retrieved passages as context to a large language model during inference.

This project is motivated by the practical need to build such a system that runs entirely on a personal computer using open-source models and libraries — making advanced AI-powered document Q&A accessible without reliance on paid cloud services or proprietary APIs. Special emphasis is placed on supporting the Vietnamese language, which is historically underserved by mainstream LLM research.

1.2 Project Objectives

The project sets out the following specific and measurable goals:

  1. Build a user-friendly web interface using Django that allows users to upload PDF documents and ask questions interactively.
  2. Integrate a multilingual embedding model to convert raw document text into high-dimensional semantic vectors that capture meaning across languages.
  3. Deploy a FAISS vector database for efficient, scalable storage of document embeddings and fast approximate nearest-neighbor retrieval at query time.
  4. Integrate the Qwen2.5:7b large language model — optimized for Vietnamese — via the Ollama local inference framework, enabling fully offline operation.
  5. Build a complete end-to-end RAG pipeline, from PDF ingestion and text splitting through embedding, indexing, retrieval, prompt construction, and final answer generation.
  6. Optimize system performance, response accuracy, and multilingual quality, with automatic language detection to ensure answers are returned in the user's language.

1.3 Project Scope

The project is scoped and constrained as follows:

  • Input format is limited to PDF documents uploaded directly by the user via the web interface.
  • Full and high-quality support for Vietnamese and English, with general support for 50+ additional languages via the multilingual embedding model.
  • Automatic detection of the user's query language, with prompt templates dynamically selected to produce a response in the matching language.
  • Entirely local deployment on a personal computer — no external cloud APIs, no data transmission, no internet required after the initial model download.
  • Exclusive use of open-source frameworks, models, and libraries: LangChain, FAISS, HuggingFace Transformers, Ollama, and Qwen2.5.
  • The system does not perform real-time web search; it answers only from the content of documents uploaded by the user.
02

Theoretical Foundations

2.1 Retrieval-Augmented Generation (RAG)

RAG is an AI architecture that enhances language model outputs by grounding them in retrieved external knowledge. Rather than relying solely on the parametric memory encoded in model weights during pre-training, a RAG system dynamically retrieves the most relevant document passages from an external knowledge base at inference time, then provides those passages as context to the language model so it can generate a precise, factually grounded answer.

The RAG architecture can be formally expressed as:

📐 RAG(x) = Generator( x, Retriever(x) )
Where x is the user's input query, Retriever(x) fetches the top-k most semantically relevant document chunks from the vector store, and Generator is the large language model that synthesizes a coherent answer conditioned on both the query and the retrieved context.

Compared to traditional keyword-based search, RAG offers several key advantages: it understands the deep semantic intent of a query, is not limited by exact term matching, can reference content from user-provided documents that the model was never trained on, and allows the knowledge base to be updated simply by re-indexing new documents — without any model retraining. This makes RAG particularly well-suited for domain-specific or document-centric question answering tasks.

💡 RAG effectively bridges the gap between the static parametric knowledge frozen inside a pre-trained model and the dynamic, domain-specific information stored in user-provided documents — making it ideal for enterprise document Q&A, legal research, academic literature review, and any scenario where answers must be grounded in a specific corpus.

2.2 Text Embeddings and Vector Databases

2.2.1 Text Embedding

Text embedding is the process of mapping natural language into fixed-dimensional numerical vectors in a continuous high-dimensional space, formally expressed as:

📐 E : Text → ℝd
Where d is the dimensionality of the output vector — commonly 384, 512, or 768 dimensions. Semantically similar texts are mapped to geometrically close vectors in this space, while semantically unrelated texts are mapped far apart.

This property enables semantic search: finding the most relevant document chunks based on conceptual meaning rather than exact keyword overlap. Similarity between vectors is measured using cosine similarity or Euclidean distance.

This project uses the paraphrase-multilingual-mpnet-base-v2 model from HuggingFace's Sentence Transformers library. Key characteristics of this model include:

🧠
MPNet Architecture

Combines masked and permuted language modeling, achieving state-of-the-art quality for multilingual semantic similarity tasks.

🌐
50+ Languages

Trained on large multilingual corpora including Vietnamese, enabling high-quality cross-lingual semantic search out of the box.

📏
768-Dimensional Output

Produces rich, high-dimensional embeddings that capture fine-grained semantic nuance suitable for document retrieval.

⚡
CPU-Friendly

Designed to run efficiently on consumer hardware without GPU requirements, enabling local deployment on personal computers.

2.2.2 FAISS — Facebook AI Similarity Search

FAISS is a high-performance open-source library developed by Facebook AI Research, designed specifically for efficient similarity search and clustering of dense floating-point vectors. In the context of a RAG system, FAISS serves as the vector database that stores all document chunk embeddings and executes fast retrieval at query time.

Key advantages of FAISS in this project:

  • Scalability: Supports fast approximate nearest-neighbor (ANN) retrieval over millions of vectors with sub-linear query time.
  • Flexible Indexing: Offers multiple index types (flat, IVF, HNSW) with different trade-offs between speed, memory, and recall accuracy.
  • GPU Acceleration: Can leverage NVIDIA GPUs for dramatic speedups when available, while also running efficiently on CPU.
  • Memory-Efficient: Supports product quantization and other compression techniques to reduce memory footprint for large indices.
  • Open Source & Free: Released under MIT license, with native Python bindings for easy integration with LangChain.

2.3 Large Language Models (LLMs)

2.3.1 Qwen2.5

Large Language Models are transformer-based neural networks pre-trained on massive multilingual text corpora spanning hundreds of billions to trillions of tokens. Through this pre-training, they develop broad language understanding capabilities including question answering, summarization, translation, reasoning, and code generation — which can then be applied to domain-specific tasks via prompting alone, without fine-tuning.

This project uses Qwen2.5:7b, developed by Alibaba Cloud. It was selected for this project due to its exceptional Vietnamese language performance — outperforming models such as Llama 3 on Vietnamese benchmarks — combined with a manageable 7-billion parameter count that runs feasibly on consumer hardware. Key characteristics:

  • Next-generation Transformer architecture with improved attention mechanisms and training efficiency.
  • 128K token context window, allowing the model to process long documents and extensive retrieved context without truncation.
  • Trained on multilingual corpus with substantial Vietnamese content, producing high-quality Vietnamese responses.
  • 7B parameters — a well-balanced scale offering strong generation quality while remaining practical for local inference on a laptop or desktop.
  • Outperforms Llama3 on Vietnamese language understanding and generation tasks.

2.3.2 Ollama

Ollama is an open-source runtime framework that simplifies the local deployment and execution of large language models on personal computers. Rather than requiring complex CUDA configurations or cloud API subscriptions, Ollama abstracts model management into a single CLI tool and exposes a clean REST API compatible with standard HTTP clients.

In this project, Ollama serves the Qwen2.5:7b model locally, providing a fully offline, privacy-preserving inference backend. Once the model is downloaded, the system operates without any internet connection — all user data and document content remains entirely on the local machine.

2.4 LangChain Framework

LangChain is an open-source Python framework purpose-built for developing applications powered by large language models. Its core design philosophy is composability: each step in an LLM pipeline is represented as a modular, interchangeable component that can be combined with others to build complex workflows with minimal boilerplate code.

The key components used in this project are:

Document Loaders

Ingest and parse documents from various file formats (PDF, DOCX, TXT, etc.) into LangChain's standardized Document objects, preserving metadata such as page numbers and source paths.

Text Splitters

Divide long documents into smaller, overlapping chunks using strategies such as recursive character splitting. Overlap between chunks ensures that semantic context is not lost at chunk boundaries.

Embeddings

Provide a unified interface for generating vector representations of text chunks, supporting HuggingFace models, OpenAI, and other providers through consistent API abstraction.

Vector Stores

Manage storage and retrieval of document embeddings. LangChain's FAISS integration handles index creation, persistence, and similarity search queries transparently.

Retrievers

Expose a standardized interface for fetching the top-k most relevant document chunks given a query embedding, decoupling retrieval logic from the rest of the pipeline.

Chains

Connect all components into an end-to-end pipeline — from raw user query, through retrieval and prompt construction, to final LLM-generated answer — using LangChain's expression language (LCEL).

2.5 Prompt Engineering

Prompt Engineering is the practice of carefully designing and optimizing the input text (prompt) provided to a language model in order to guide it toward producing accurate, relevant, and appropriately formatted outputs. In a RAG system, the prompt is the critical bridge between the retrieved document context and the final generated answer — a poorly designed prompt can cause the model to hallucinate, ignore the context, or respond in the wrong language.

An effective RAG prompt in this project is structured around four elements:

📋
Instruction

Directs the model on how to behave: use only the provided context, admit uncertainty if the answer is not present, and keep responses concise (3–4 sentences).

📄
Context

The concatenated top-k document chunks retrieved by FAISS, providing the factual grounding from which the model must derive its answer.

❓
Question

The user's original query, preserved verbatim so the model understands exactly what is being asked without paraphrase distortion.

🔒
Constraint

Explicit formatting and length constraints that prevent over-verbose outputs and ensure the model responds strictly in the detected user language.

A key design feature of this system's prompt engineering is automatic language detection. The system inspects the user's query for Vietnamese diacritical characters; if detected, a Vietnamese-language prompt template is selected, instructing the model to respond entirely in Vietnamese. Otherwise, the English prompt template is used. This ensures language-appropriate responses without requiring the user to specify their preferred language explicitly.

03

System Design

3.1 System Architecture

The system is designed following a clean multi-layer architecture that separates concerns across four distinct tiers, each with a well-defined responsibility. This design promotes modularity, independent replaceability of components, and straightforward extensibility as the system evolves.

🌐 Presentation LayerDjango Web Interface · File Upload · Answer Display
↓
⚙️ Application LayerLangChain · RAG Pipeline · Prompt Engineering
↓
🗄️ Data LayerFAISS Vector Store · MPNet Embeddings (768-dim)
🤖 Model LayerQwen2.5:7b via Ollama · Local LLM Inference

The Presentation Layer is built with Django and handles all user-facing interactions: PDF file uploads, question input, and answer display. The Application Layer orchestrates the entire RAG workflow via LangChain, including document processing, prompt construction, and response generation. The Data Layer manages the FAISS vector index and the multilingual embedding model. The Model Layer runs Qwen2.5:7b locally through Ollama, receiving prompts and returning generated answers entirely offline.

3.2 Data Flow

The system operates through two distinct and sequential processing flows:

📥 Document Ingestion Flow
PDF Upload → Text Extraction → Chunk Splitting → Embedding → FAISS Index
🔎 Query Processing Flow
User Query → Lang Detect → Top-k Retrieve → Prompt Build → LLM Answer

During document ingestion, uploaded PDFs are parsed into raw text, split into overlapping chunks of up to 1,000 characters (with 100-character overlap to preserve cross-boundary context), converted into 768-dimensional embeddings, and stored in the FAISS index. During query processing, the user's question is embedded using the same model, the top-3 most similar chunks are retrieved from FAISS, assembled into a structured prompt together with the language-appropriate instruction template, and passed to Qwen2.5:7b via Ollama for answer generation.

3.3 Component Description

Document Loader

Uses PDFPlumberLoader from LangChain Community to read uploaded PDF files. PDFPlumber accurately extracts text content including tables and multi-column layouts, preserves page structure and layout information, and handles complex PDFs more robustly than simpler text extractors.

Text Splitter

Uses LangChain's RecursiveCharacterTextSplitter with a chunk size of 1,000 characters and a 100-character overlap. The recursive strategy attempts to split on paragraph boundaries first, then sentence boundaries, then words — minimizing semantic fragmentation. The overlap window ensures that information near chunk boundaries is captured by adjacent chunks.

Embedding Model

Uses paraphrase-multilingual-mpnet-base-v2 via LangChain's HuggingFaceEmbeddings wrapper. Embeddings are computed on CPU with normalized output vectors, producing 768-dimensional representations that support semantic search across Vietnamese, English, Chinese, and 50+ other languages.

Vector Store (FAISS)

All document chunk embeddings are stored in a FAISS flat index built via LangChain's FAISS integration. At query time, a retriever is configured with search_type="similarity" and k=3, returning the three most semantically relevant chunks to serve as context for the language model.

LLM Integration

Qwen2.5:7b is served locally via the Ollama runtime, accessed through LangChain's Ollama integration. The model receives a fully constructed prompt containing the retrieval context and the user's question, and generates a concise, grounded answer. All inference runs on the local machine — no data leaves the user's device.

Prompt Engine

The prompt engine automatically detects the language of the user's query by scanning for Vietnamese diacritical characters. Based on the detection result, it selects the appropriate language-specific prompt template — Vietnamese or English — and injects the retrieved context and user question before dispatching to the LLM.
